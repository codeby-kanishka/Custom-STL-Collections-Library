# Design Journal - Day 1

## Section 1 — Specific Bug

Runtime Issue:

DynamicArray failed while handling non-primitive data types like std::string.

The problem occurred because memory was allocated using malloc, which only provides raw memory and does not automatically call constructors.

Objects requiring initialization were not created correctly.


## Section 2 — Failed Attempt

Initially, I used malloc to allocate memory and directly assigned values into that memory.

T* arr = (T*)malloc(sizeof(T) * capacity);

This worked with primitive data types because they do not require special initialization.

However, for objects like string, the constructor was never called, so the internal object state was not created properly.

The solution was using placement new:

new(&arr[index]) T(value);

This allowed object construction inside already allocated memory.

## Section 5 — Learning Reflection

Initially, I considered memory allocation and object creation as the same process.

After implementing manual memory management, I understood that allocating memory only reserves space, while constructors create usable objects.

I learned why containers separate allocation and construction, and why placement new is required when using malloc with generic types.

I also understood the importance of cleaning partially created objects if an error occurs during resizing.
