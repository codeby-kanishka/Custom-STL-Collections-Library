# Design Journal - Day 2

## Section 1 — Specific Bug

Design Issue:

LinkedList required efficient insertion at the end and HashMap needed a way to handle multiple keys mapping to the same bucket index.

Initially, insertBack operation required traversing the complete LinkedList every time, and HashMap required collision handling for duplicate bucket indexes.


## Section 2 — Failed Attempt

Initially, the LinkedList only maintained a head pointer.

For inserting an element at the end, traversal was required:

head -> node -> node -> last node

This made every insertBack operation traverse the entire list.

The solution was adding a tail pointer that always stores the address of the last node, allowing direct insertion at the end.

While implementing HashMap, initially storing values directly using the calculated index was considered:

index = hash(key)

However, different keys can generate the same bucket index and cause collisions.

The solution was implementing separate chaining where every bucket contains a LinkedList of key-value pairs.

## Section 5 — Learning Reflection

Initially, I implemented LinkedList operations focusing only on node connections.

After improving insertBack, I understood how maintaining additional information like a tail pointer can reduce unnecessary traversal and improve the design.

While implementing HashMap, I understood that hash values do not always guarantee unique positions and collisions are expected.

I learned how two data structures can work together, where DynamicArray manages buckets and LinkedList handles collisions through separate chaining.
