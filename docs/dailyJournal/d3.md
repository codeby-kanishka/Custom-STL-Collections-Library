# Design Journal - Day 3

## Section 1 — Specific Bug

Design Issue:

HashMap required proper copy behavior because it internally manages dynamically allocated data structures.

The DynamicArray interface also needed improvement to support read-only access and clear element modification.


## Section 2 — Failed Attempt

Initially, HashMap objects relied on default copying behavior.

The default copy operation could copy internal data directly instead of clearly defining how the container should manage its own copy.

Since HashMap stores buckets containing LinkedLists, proper copying behavior was required.

The solution was implementing the Rule of Three:

- Destructor
- Copy constructor
- Assignment operator

The template implementation was also separated into .h and .tpp files to improve code organization.

For DynamicArray, initially only one get method existed:

T& get(int index);

This allowed accessing and modifying values but did not support const objects.

The solution was adding:

- const get() for read-only access
- set() method for controlled modification

## Section 5 — Learning Reflection

Initially, I thought that copying an object containing other data structures would automatically create a completely independent copy.

While implementing HashMap, I understood that classes managing dynamic memory need their own copy logic to prevent memory ownership problems.

I learned how the Rule of Three helps safely manage resources using destructor, copy constructor, and assignment operator.

I also understood const correctness in C++ and why containers provide separate read-only and modification methods.

This improved my understanding of designing reusable container APIs instead of only focusing on implementation.
